# Internationalisation (i18n) Architecture

## Overview

The application supports two languages: **English** (default) and **Polish**. Language preference is stored as a cookie (`locale=en|pl`) and applies to all user-visible UI text. User-entered content (case titles, document names, contract text) is never translated — only UI strings.

## Library

**`next-intl`** — chosen because it is the only mature i18n library that works across both server and client components in the Next.js 15 App Router without requiring URL-based route restructuring (`/en/...`, `/pl/...`).

## Configuration

| File | Role |
|------|------|
| `i18n/request.ts` | Reads the `locale` cookie; loads the corresponding messages file |
| `messages/en.json` | English message strings (source of truth) |
| `messages/pl.json` | Polish message strings |
| `next.config.mjs` | Wrapped with `withNextIntl` plugin |
| `src/app/layout.tsx` | Async root layout; injects `NextIntlClientProvider` and sets `<html lang>` dynamically |
| `src/app/api/locale/route.ts` | `POST /api/locale` — sets the `locale` cookie |

## Language Switching

The language toggle is in the `AppSidebar` footer (alongside the theme toggle). Clicking it:
1. `POST /api/locale` with the new locale
2. Sets `locale` cookie (1-year expiry, path `/`)
3. `window.location.reload()` — required because server components render with the locale at request time

A hard reload is intentional and acceptable: language changes are infrequent, and all server-side components must re-render with the new locale.

## Message File Structure

Messages are namespaced by feature area. All namespaces must exist in both `en.json` and `pl.json` with identical keys.

```
Common        — shared labels: Save, Cancel, Delete, Edit, Back, Loading…
CaseStatuses  — case status display names (new, intake, analysis, …)
CaseTypes     — case type display names (civil, criminal, …)
DocCategories — document category labels
Sidebar       — navigation labels
Auth          — login, register, invite pages
LegalHub      — Legal Hub feature (cases, templates, wizard, blueprints, deadlines, …)
Contracts     — contracts and obligations
Documents     — document library, analysis, policies, ask
Settings      — user settings, org settings, member management
Admin         — super admin panel
```

## Usage in Components

**Server components:**
```ts
import { getTranslations } from 'next-intl/server';

export default async function MyPage() {
  const t = await getTranslations('LegalHub');
  return <h1>{t('cases')}</h1>;
}
```

**Client components:**
```ts
'use client';
import { useTranslations } from 'next-intl';

export function MyComponent() {
  const t = useTranslations('Common');
  return <button>{t('save')}</button>;
}
```

**Dynamic keys (e.g. status labels from DB values):**
```ts
const t = useTranslations('CaseStatuses');
const label = t(status); // t('new') → "New" or "Nowa"
```

## Bilingual Requirement for New Features

> **All new features must be built bilingually from the start.**

Hardcoding UI strings directly in component code is not permitted. Every string that a user can read must have a key in both `messages/en.json` and `messages/pl.json`.

**Checklist for every new feature:**
- [ ] All button labels, headings, placeholders, error messages, and toast messages use `t('Key')` — never hardcoded strings
- [ ] New translation keys are added to **both** `en.json` and `pl.json` before the PR is merged
- [ ] New constants that produce display labels (e.g. status maps, type maps) use the `CaseStatuses`/`CaseTypes` pattern — translated via `t()` at the call site, not as raw strings in constants
- [ ] Any new namespace added to `en.json` is added to `pl.json` with all keys present

Failure to add keys to both files will cause a runtime error in next-intl (missing message warning in development, fallback to key name in production).

## Default Language

**English** is the default for unauthenticated users and users without a `locale` cookie. The cookie is set when the user actively toggles the language.

## Out of Scope

- URL-based locale routing (`/en/...`, `/pl/...`)
- Translation of user-entered content (case titles, document names, contract body text)
- Translation of AI-generated output (chat responses, template suggestions)
- Translation of API error messages returned from the backend (displayed in toasts as-is)
- Additional languages beyond English and Polish
